## 二、详细流程图

### 阶段一：任务提交

```
用户请求
  │
  ▼
POST /v1/workflow/run-async  或  POST /v1/workflow/run-scene-async
  │
  ├─ [场景接口] 查询 scene → 获取 workflow_id、is_public_scene
  │
  ├─ 自动注入 token / tenant_name / site_host 到 parameters
  │
  ├─ 调用 workflowService.SubmitCozeWorkflow()
  │     │
  │     ├─ 创建 CozeWorkflowRuns 记录 (status=pending)
  │     │
  │     ├─ 创建 Asynq Task (payload: TenantID, RunID, WorkflowID, Parameters...)
  │     │
  │     └─ 入队 queue="coze", MaxRetry=3, Timeout=5min
  │           └─ 更新 run.asynq_task_id
  │
  └─ 立即返回 { run_id, status:"pending", message:"任务已提交" }
```

### 阶段二：队列 Worker 执行

```
Asynq Worker 消费 "coze" 队列
  │
  ▼
HandleCozeSingleExecute(ctx, task)
  │
  ├─ 解析 payload (TenantID, RunID, WorkflowID, Parameters...)
  │
  ├─ 获取租户 DB
  │
  ├─ 查询 CozeWorkflowRuns 记录
  │     └─ 若 status=cancelled → 直接返回 nil（不执行）
  │
  ├─ 更新 status=running, start_time=now
  │
  ├─ callCozeAPI()
  │     ├─ 获取 AccessToken（公共场景用系统默认，否则用租户配置）
  │     ├─ POST /v1/workflow/run  (is_async=true)
  │     └─ 返回 { code, execute_id, debug_url, ... }
  │
  ├─ [成功] 解析 execute_id
  │     │
  │     ├─ 解析 parameters["token"] → 获取 userID / username
  │     │
  │     ├─ 创建 CozeWorkflowExecuteContext 记录
  │     │     (execute_id, run_id, token, need_fetch_result, scene_id...)
  │     │
  │     ├─ 判断 need_fetch_result:
  │     │     ├─ true  → status 保持 running（等待 Cron 查询结果）
  │     │     └─ false → status 直接改为 success
  │     │
  │     └─ 更新 run (status, response_body, end_time, execute_id, debug_url)
  │
  └─ [失败] 更新 status=failed, error=err.Error()
           └─ 返回 err → Asynq 触发重试（最多3次）
```

### 阶段三：Cron 结果轮询（核心延迟所在）

```
Asynq Scheduler
  │  @every 5s（config: result-fetch-interval）
  ▼
HandleCozeResultFetcher(ctx, task)
  │
  ├─ 从主 DB 查询所有租户配置
  │
  └─ 遍历每个租户（串行）
        │
        ├─ 获取租户 DB
        │
        ├─ 查询 status=running AND updated_at < now-2s 的记录
        │     ⚠️ 注意：刚变为 running 的记录需等 2 秒才会被扫描
        │
        └─ 遍历每条 Running 记录（串行）
              │
              ├─ 查询 CozeWorkflowExecuteContext (by run_id)
              │     └─ 若 need_fetch_result=false → 跳过
              │
              ├─ fetchCozeResult()  ← 同步 HTTP 请求（30s 超时）
              │     ├─ GET /v1/workflow/get_run_history
              │     │   ?execute_id=xxx&workflow_id=xxx
              │     │
              │     ├─ data 为空 → 返回 "Running"（Coze 侧还未入库）
              │     ├─ execute_status="Success" → 返回 "Success", output
              │     ├─ execute_status="Fail"    → 返回 "Fail", errorMsg
              │     └─ 其他 → 返回 "Running"
              │
              ├─ [Success] updateRunStatusSuccess()
              │     └─ pipelineSvc.ProcessCozeResult()  ← 触发后续 Pipeline
              │
              ├─ [Fail]    updateRunStatusFailed()
              │     └─ pipelineSvc.ProcessCozeResult()
              │
              └─ [Running] 检查超时
                    └─ elapsed > result-fetch-timeout(300s)
                          └─ updateRunStatusTimeout() → status=failed
```

### 阶段四：用户轮询查询结果

```
用户轮询
  │
  ▼
POST /v1/workflow/results  { run_ids: [xxx] }
  │
  ├─ GetCozeWorkflowResults(tenantID, runIDs)
  │     └─ SELECT * FROM coze_workflow_runs WHERE id IN (...)
  │
  └─ 返回每条记录的当前状态：
        ├─ pending   → "任务排队中，请稍后查询"
        ├─ running   → "任务执行中，请继续轮询"
        ├─ success   → result = response_body
        ├─ failed    → error = 错误信息
        └─ cancelled → "任务已取消"
```

## 三、时序图（最坏情况延迟分析）

```
时间轴
  │
T+0ms      用户提交请求
  │
T+50ms     API 返回 run_id，run.status = pending
  │
T+~200ms   Asynq Worker 消费任务
           调用 Coze API (is_async=true) ← 很快，几百ms
           run.status = running，updated_at = T+200ms
  │
  │         ⚠️ 延迟点 1：等待 updated_at 满足 < now-2s
  │
T+2200ms   updated_at 满足 2s 冷却条件
  │
  │         ⚠️ 延迟点 2：等待下一个 Cron 触发（最坏等 5s）
  │
T+~5000ms  Cron 触发，但若前面还有其他租户/记录在串行处理
  │
  │         ⚠️ 延迟点 3：串行 HTTP 请求（每条记录 ~500ms）
  │
T+~6000ms  fetchCozeResult() 调用 Coze get_run_history
  │
  │         ⚠️ 延迟点 4：Coze 侧刚提交可能 data 为空
  │         若 data 为空 → 返回 Running → 浪费一个 Cron 周期
  │
T+~11000ms 下一个 Cron 周期，再次查询，终于拿到 Success
           run.status = success
  │
T+~11500ms 用户轮询 /results，终于拿到成功结果

总延迟：最坏情况约 10~15 秒（Coze 工作流本身执行只需 1~3 秒）
```

## 四、关键配置参数

| 配置项 | 当前值 | 作用 |
|--------|--------|------|
| `result-fetch-interval` | 5s | Cron 扫描间隔 |
| `result-fetch-timeout` | 300s | 单条记录最大等待时间 |
| `timeout` | 300s | 调用 Coze API 的 HTTP 超时 |
| `updated_at < now-2s` | 硬编码 2s | Running 记录冷却时间（避免重复扫描） |
| Asynq MaxRetry | 3次 | 调用 Coze 失败时的重试次数 |
| Asynq Timeout | 5min | 单个 Worker 任务最大执行时间 |

## 五、已识别的延迟根因

### 根因 1：Cron 固定间隔 + 2s 冷却 = 最少 2~7s 延迟

```
run 变为 running (T+200ms)
    ↓ 等 2s 冷却
T+2200ms 才能被扫描
    ↓ 等下一个 Cron（最坏 5s）
T+7200ms 才被处理
```

### 根因 2：串行处理多租户多记录

`HandleCozeResultFetcher` 对所有租户、所有 Running 记录**逐条串行**调用 Coze HTTP 接口。
若有 10 条 Running 记录，每条耗时 500ms，则整个 Cron 任务耗时 5s，后面的记录要等很久。

### 根因 3：Coze 侧异步延迟导致 data 为空

Coze 的 `get_run_history` 在工作流刚提交后短时间内可能返回空 data，
此时代码返回 `"Running"` 状态，浪费一整个 Cron 周期（5s）。

## 六、优化建议

### 方案一：快速见效（调整配置）

```yaml
coze:
  result-fetch-interval: 1   # 从 5s 改为 1s
```

同时将代码中 `updated_at < now-2s` 的冷却时间改为 500ms：

```go
// coze_result_fetcher.go 第94行
timeThreshold := time.Now().Add(-500 * time.Millisecond)  // 原来是 -2s
```

### 方案二：根本解决（专属延迟任务）

在 `HandleCozeSingleExecute` 拿到 `execute_id` 后，立即投递一个**专属的延迟查询任务**：

```
HandleCozeSingleExecute 成功 → 投递 CozeResultFetchTask(run_id, execute_id)
                                  延迟 2s 后执行
                                  若结果还是 Running → 再延迟 3s 重新投递
                                  直到 Success/Fail/Timeout
```

优点：
- 每条记录独立处理，互不阻塞
- 不依赖全局 Cron 扫描
- 响应延迟从 10s+ 降低到 2~4s
